# AgentDevClaw 比赛定位与产品化转向报告

## 1. 背景

`AgentDevClaw` 当前并不是一个传统意义上的垂直行业应用，而是一个围绕 Agent 装配、调试、交付和能力沉淀展开的 Web 化工作台。它的核心思路不是再做一个重型框架，也不是做一个只能配置身份提示词的弱能力产品，而是面向企业、机构内部懂业务、也懂一点技术的人，帮助他们更快地把智能体相关想法落地。

项目当前已经形成了一些比较明确的基础骨架：

- 基于 Web 的独立工作台与调试界面
- `Feature Repository` 作为能力资产浏览与管理入口
- `Feature Creator` 作为 Feature 生产与打包入口
- `Agent Creator` 作为 Agent 装配与项目开发入口
- 工作空间状态、过程记录、项目文档集等沉淀机制

在产品理解上，这个项目与以下几类路线形成对比：

- 与 LangChain 一类重型编排方案相比，强调轻量、直接、少状态机负担
- 与“直接让编程 Agent 管 MCP / skills”的无治理方式相比，强调受控装配与可追踪
- 与纯图形化流程产品相比，强调保留真实编程与扩展能力
- 与只允许配置提示词身份的商业产品相比，强调不仅能配角色，还能配能力、工作空间和交付路径

换句话说，项目的价值不在单一算法创新，而在于：把 Agent 落地从零散试错，收束为可理解、可装配、可调试、可沉淀的工作流。

## 2. 当前面临的问题

在比赛语境下，项目遇到的核心矛盾并不是“有没有价值”，而是“如何被正确理解和归类”。

一方面，普通“软件应用与开发”大类下面的几个常见小类与当前项目存在天然错位：

- 它不是典型的算法设计作品，亮点不在单一模型或算法突破
- 它不是传统管理信息系统，管理对象不是财务、人事、库存等经典业务数据
- 它不是移动端产品
- 它虽然是 Web 产品，但普通 `Web 应用与开发` 小类对公网可访问站点有明确要求，这对当前项目形成额外形式门槛

另一方面，项目如果继续按“Agent 框架”或“开发底座”去讲，会有明显风险：

- 评委容易把它理解成偏底层工具，而不是可交付的软件系统
- 很难直接映射到“解决真实场景问题”的比赛叙事
- 会把项目优势讲窄，忽略它其实已经具备较强的产品化工作台雏形

因此，这次讨论的重点不是“项目该不该继续做”，而是“项目该以什么身份出现”。

## 3. 比赛方向判断

经过讨论，当前已经形成比较明确的判断：

### 3.1 不优先走普通算法类叙事

项目不应主打算法设计与应用。原因很明确：

- 现阶段核心价值不是算法创新
- 比赛优势在产品形态、工作台结构、能力装配和应用落地
- 如果硬往算法类靠，会削弱项目最有竞争力的部分

### 3.2 普通 Web 小类在形态上匹配，但形式门槛偏高

如果只看产品形态，`AgentDevClaw` 作为 B/S 结构的 Web 工作台，的确很适合往 `Web 应用与开发` 靠拢。但普通 Web 小类要求提供互联网可访问地址，这会把问题从“定位”变成“部署与审核形式要求”。

因此，普通 Web 小类不是不能报，而是对当前阶段来说不是最顺的一条路。

### 3.3 当前最合适的方向是 AI 应用专项赛

目前最合理的选择，是转向“软件应用与开发专项赛”中“基于人工智能的多元场景应用设计与开发”这一方向。

这个方向与项目当前状态更加匹配，原因包括：

- 它强调的是“基于人工智能技术的平台、模型或组件，设计并开发解决真实场景问题的应用系统”
- 它接受 B/S、C/S 或移动终端形态
- 它更看重真实场景与完整演示，而不是死卡某种传统软件分类
- 它允许基于已有 AI 底层能力做应用级产品，不要求参赛作品必须是从零训练一个算法系统

这意味着，`AgentDevClaw` 不必被硬塞进一个不自然的小类，而可以在更合理的口径下被表述为：

一个面向多元真实场景的 AI 应用设计、装配、调试与交付平台。

## 4. 产品定位的收束

围绕专项赛方向，项目的产品定位也已经有了比较明确的收束：

### 4.1 不再主打“通用 Agent 框架”

继续用“框架”“底座”“runtime”“通用 Agent 开发引擎”来描述项目，会把项目带回技术底层叙事，不利于比赛表达，也不利于产品化理解。

因此，项目不应继续把自己当成“另一个 Agent 框架”来讲。

### 4.2 不走“任意自由开发任何 Agent”的主叙事

理论上，项目确实支持进一步开发 Agent 和 Feature，但当前产品状态并不适合把“无限自由开发”作为第一卖点。原因在于：

- 工作空间结构存在耦合
- 用户自定义页面与表单的开放成本较高
- 如果主打完全自由开发，评委和用户会天然期待更强的低代码/可视化/自定义能力

因此，自由开发能力应被保留为平台深层能力，而不作为第一层定位。

### 4.3 现阶段主打“高频场景解决方案工作空间”

最终收束下来的方向是：

- 主打一组面向高频实际需求的应用型工作空间
- 每个工作空间代表一个相对完整的 AI 解决方案入口
- 这些解决方案是通用的、可套路化的，不要求深耕某一个垂直行业
- 在此基础上，再通过 `Agent Creator` 和 `Feature Creator` 证明平台具备进一步扩展与升级能力

这意味着产品对外的第一层不是“你可以开发任何 Agent”，而是：

“我们已经为一批常见真实场景预设了可直接装配和使用的智能解决方案工作空间。”

而 Creator 相关能力属于第二层：

“如果这些预设还不够，还可以继续升级、扩展和项目化开发。”

## 5. 对“场景”的最终理解

本轮讨论里最重要的一个判断是：专项赛并不强迫项目去深挖某一个狭窄行业，也不要求一定选择农业、医疗、工业这样的重行业壳子。

项目更适合选择的是“场景簇”而不是“单点行业深坑”。

所谓场景簇，是指：

- 它们都对应真实、重复、高频的任务需求
- 它们不依赖某一个垂直领域的复杂专业知识
- 它们可以在企业、学校、机构、团队等多种组织环境下复用
- 它们适合被收束成一套“预设能力 + 表单参数 + 启动实例 + 过程沉淀”的解决方案模板

这种理解与项目现状高度匹配，因为它允许项目：

- 不必强行变成某个行业专用系统
- 也不必停留在纯技术平台层
- 而是落在“多元真实场景的 AI 应用设计与开发”这个中间层

## 6. 当前已经形成的产品化思路

在产品化推进上，目前形成的整体思路可以概括为三层结构：

### 6.1 管理型工作空间

这类工作空间不直接对外承担具体应用场景，而是承担平台级管理职责。当前主要包括：

- `Feature Repository`
- 可能的整体总览入口

它们的职责是让平台显得有资产管理和能力组织能力，而不是只有几个散落的 Agent。

### 6.2 元能力工作空间

这类工作空间主要承担“构建”和“升级”职责。当前主要包括：

- `Feature Creator`
- `Agent Creator`

它们是平台性的证据，说明项目不只是若干 demo，而是具备从预设方案走向更深扩展的能力。

### 6.3 应用型工作空间

接下来最需要补强的是这一层。也就是把当前已有的应用型 Agent 或机器人，包装为更完整的“解决方案工作空间”，并继续补充若干场景型入口。

这一层将成为比赛展示的主角。

## 7. 对应用型工作空间的判断

针对应用型工作空间，当前已经有几个比较明确的结论。

### 7.1 核心不是“做更多 Agent”，而是“做更像解决方案的工作空间”

如果只是继续往左侧列表里加更多 Agent，项目仍然会像一个杂乱的 bot 集合。

更合理的做法是：

- 给每个应用入口一个明确场景身份
- 增加启动表单、过程记录、最近实例、配置保存等结构
- 让它看起来像一个完整的应用型工作台

### 7.2 主打数量可以有限，但必须足够像样

比赛层面，不需要把每一个场景都做深，但至少需要几类“主打解决方案工作空间”足够完整，能够支撑展示与叙事。

因此当前思路是：

- 主打 4 到 5 个代表性工作空间
- 其余少量补充，形成多场景版图
- 不追求每个都很深，但要让平台看起来具备明显的场景覆盖能力

### 7.3 场景工作空间要强调“可直接用”

当前更合适的表达不是：

- “帮助你自由开发某类 Agent”

而是：

- “我们已经为一类高频场景预设好了可直接装配和使用的工作空间”

然后在第二层再说明：

- “如果需要更深定制，还可以升级到 Creator 路径”

## 8. 对 Creator 的最终角色判断

讨论中还形成了一个比较稳定的认识：`Feature Creator` 和 `Agent Creator` 不应该成为比赛演示的主角，但它们必须存在，而且必须看起来专业。

它们的意义在于：

- 证明平台不是单纯的固定 demo 集合
- 证明平台具备从预设到扩展的升级能力
- 证明平台在能力资产和项目化开发上有更深一层的工作机制

但它们不应承担“完全开放给用户自由设计所有工作空间 UI 和逻辑”的叙事，因为当前阶段这既不现实，也没有必要。

更稳的定性是：

- 应用型工作空间负责快速上手
- Creator 工作空间负责高级配置、扩展与升级

这种分层既贴合当前实现状态，也有利于比赛表达。

## 9. 关于“预设”与“开发能力”的平衡

当前已经明确，不应把项目简单定义成二选一：

- 要么是“现成 Agent 列表”
- 要么是“通用 Agent 开发平台”

更准确的表达应当是：

- 第一层，平台提供一组面向高频场景的预设解决方案工作空间
- 第二层，平台支持在这些预设基础上继续升级和扩展

也就是说，项目主打的是：

“从可直接使用，到可进一步定制，再到可项目化开发”的连续路径。

这个判断非常关键，因为它既规避了当前阶段自定义工作空间成本高的问题，也保住了项目的平台属性。

## 10. 当前约束与现实边界

这次讨论里还有一个很重要的共识：比赛阶段不应追求把所有能力做成长期产品的理想形态，而应在真实约束下做出最优展示。

当前约束包括：

- 时间有限
- 不能围绕某个垂直行业重新做一整套深应用
- 工作空间系统本身存在一定耦合
- 用户级自由自定义 UI / 表单的成本偏高
- 某些经典流程需求如果要求完整自定义实现，工作量会迅速上升

因此，当前更务实的选择是：

- 不去追求彻底的通用性
- 不去追求对任意场景的完全开放开发
- 而是优先把高频、易懂、容易展示的场景做成套路化解决方案

这并不意味着项目方向错误，反而意味着项目在比赛阶段要优先突出最有价值、最容易被理解的一面。

## 11. 演示与叙事上的总体方向

从比赛展示角度，后续的整体叙事已经逐渐清晰：

- 首页与官方工作空间提供统一入口
- 若干应用型工作空间体现平台对真实场景的覆盖
- `Feature Repository` 体现能力资产组织
- `Agent Creator` / `Feature Creator` 体现升级与扩展能力
- 最终让评委感知到，这不是单点应用，也不是技术底座，而是一套面向多场景 AI 应用落地的工作台系统

这种叙事比“这是一个 Agent 框架”更强，也比“这是某个很窄场景的小应用”更真实。

## 12. 当前阶段的最终结论

截至本次讨论，已经可以把阶段性结论收束为以下几点：

1. 当前项目最适合转向“基于人工智能的多元场景应用设计与开发”专项赛方向。
2. 项目不应继续主打“通用 Agent 框架”或“自由开发一切 Agent”的叙事。
3. 项目应被表述为一个面向多元真实场景的 AI 应用设计、装配、调试与交付平台。
4. 产品第一层应主打若干高频场景解决方案工作空间，而不是单纯的 Agent 列表。
5. `Feature Creator` 与 `Agent Creator` 属于平台的元能力层，用于证明扩展与升级能力，而不是替代应用型工作空间。
6. 场景表达应采用“场景簇”思路，强调真实、高频、通用、可套路化，而不是强行深耕某个垂直行业。
7. 比赛阶段不追求极致完备，而追求叙事清晰、结构完整、演示顺畅和感知价值明确。

## 13. 这份报告的用途

这份文档的目标不是给出具体页面设计，而是统一当前团队对项目局面的理解，避免后续推进时再次回到以下误区：

- 把项目重新讲回底层框架
- 过度纠结传统赛道分类而忽略专项赛更匹配
- 试图强行深挖某一个不自然的行业方向
- 在比赛阶段过早追求完全自由的工作空间开发能力

后续具体功能设计、工作空间规划、预设方案补强，都应建立在本报告所确认的整体判断之上。
