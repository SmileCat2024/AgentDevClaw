# 生成 GROUP.md

你的任务是为当前群聊生成一份 GROUP.md 文档。这份文档的读者是**你自己（群聊管理员/协调者）**，目的是让你在后续的群聊调度中能够准确理解消息含义、正确识别任务归属、理解项目术语。

## 第一步：收集信息

依次调用以下工具收集素材：

1. **`gc_scan_workdir`** — 扫描群聊绑定的工作目录，了解项目结构
2. **`gc_messages`** — 查看最近的群聊历史（建议 limit=50），了解团队实际在讨论什么、怎么协作的
3. **`gc_status`** — 查看群聊中可用的 Agent 身份及其能力

## 第二步：生成 GROUP.md

基于收集到的信息，写一份 GROUP.md。**记住你的读者是协调者，不是执行者。**

### 应该包含的内容

- **项目概述**：这个群在做什么？目标是什么？（1-3 句话）
- **关键术语**：项目中出现的专有名词、缩写、工具名、服务名，附简短解释。这是最重要的部分——管理员需要靠它理解群聊消息。
- **团队分工**：群里有哪些角色（用户 + 各 Agent），各自负责什么。这帮助管理员判断任务应该派给谁。
- **任务边界**：什么样的请求属于哪个 Agent 的职责范围。如果存在容易混淆的边界（如 A 和 B 的职责有重叠），明确说明。
- **协作约定**：团队有没有特定的沟通模式或工作流约定（例如"需求先讨论再执行"、"结果需要审核"等）。

### 不应该包含的内容

- 构建/测试/lint 命令——管理员不需要执行这些
- 代码架构细节、文件级结构——管理员不需要读代码
- API 文档、接口定义——太底层
- 编码规范、提交规范——这是执行者的事
- 任何只有写代码的人才需要的信息

### 写作原则

- **每一行都要通过这个检验**：如果删掉这一行，管理员在调度消息时可能出错吗？如果不会，就删掉。
- **聚焦"理解"而非"执行"**：管理员需要的是"这个项目在做什么、谁在做什么"，不是"代码怎么写"。
- **术语解释是核心价值**：如果群里有人说"把 X 接到 Y 上"，管理员需要知道 X 和 Y 是什么，才能正确路由消息。
- **群聊的场景不限于编码**：可能是研究、写作、数据分析、项目管理等任何需要多 Agent 协作的事务。根据实际场景调整内容。

## 第三步：保存并通知

1. 调用 **`gc_save_group_md`** 保存生成的 GROUP.md
2. 调用 **`gc_reply`** 告知用户 GROUP.md 已生成，简述其中包含的关键信息（不要粘贴全文）
